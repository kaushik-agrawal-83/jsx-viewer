# Phase 3 — Release Management

## Status
> ✅ COMPLETE  
> Agent: update → 🔄 IN_PROGRESS → ✏️ CODE_COMPLETE → 🧪 TESTING_COMPLETE → ✅ COMPLETE

## Goal
Automated versioning, changelog generation, and GitHub Releases via release-please.
Every phase wave ends with a conventional commit that rolls into a versioned release.
No manual version bumping or changelog editing — ever.

## Demo Checkpoint
- [ ] Merge a `feat(...)` commit to `main` → GitHub Actions runs release-please → opens a Release PR
- [ ] Merge the Release PR → GitHub creates a tagged release (`v0.1.0`) with auto-generated CHANGELOG
- [ ] Next `fix(...)` commit → release-please updates the open Release PR (patch bump)
- [ ] `git tag` shows `v0.1.0` after merge; `CHANGELOG.md` has correct entries

### GitHub Actions — release-please Workflow

- [ ] `.github/workflows/release-please.yml`:
  ```yaml
  name: Release Please

  on:
    push:
      branches: [main]

  permissions:
    contents: write
    pull-requests: write

  jobs:
    release-please:
      runs-on: ubuntu-latest
      steps:
        - uses: googleapis/release-please-action@v4
          with:
            config-file: release-please-config.json
            manifest-file: .release-please-manifest.json
  ```

### Conventional Commit Convention (per phase)

Every phase ends with **one squash commit** to `main` in this format:
```
feat(phase-N): <short description of what the phase delivers>

<bullet-point summary of changes — used verbatim in CHANGELOG>

- Add <thing 1>
- Add <thing 2>
- Wire <thing> into <thing>
```

**Phase commit examples:**
| Phase | Example commit |
|-------|----------------|
| 2 | `feat(phase-2): add pre-commit hooks and GitHub Actions CI` |
| 3 | `feat(phase-3): add release-please automated release management` |
| 4 | `feat(phase-4): add JSX transpile pipeline with Babel standalone` |
| 5 | `feat(phase-5): add sidebar with drag-drop file opening` |
| 6 | `feat(phase-6): add preview pane with ErrorBoundary and Tailwind isolation` |
| 7 | `feat(phase-7): add tabbed interface with session persistence` |
| 8 | `feat(phase-8): add split-view pane with draggable divider` |

| 10 | `feat(phase-10): migrate to Tauri v2 native macOS app` |

| 12 | `feat(phase-12): production hardening, logging, and README` |
| 13 | `feat(phase-13): add file watcher hot-reload (opt-in)` |

**Bug fix within a phase:**
```

```

**Breaking change (triggers major version bump):**
```
feat(phase-N)!: <description>

BREAKING CHANGE: <what changed and why>
```

## Notes
- release-please bot needs `contents: write` + `pull-requests: write` permissions in the workflow. These are scoped to the Actions bot token (`github.token`) — no PAT needed.
- `bump-minor-pre-major: true` — pre-1.0 `feat` commits bump minor (0.0.0 → 0.1.0), not major. This is intentional: the app is pre-release.
- release-please reads commit history since the last tag. If commits before Phase 3 aren't tagged, it will lump them all into the first release — acceptable.
- Each agent completing a phase should end with a single `feat(phase-N): ...` squash commit. Do not create multiple commits per phase — one commit, one release entry.
- The Release PR ("chore: release vX.Y.Z") is auto-created/updated by the bot. Humans merge it when ready to cut a release. It does NOT need to be merged after every phase — release when the phase represents a meaningful checkpoint.
